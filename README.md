# Ledger Service

A small Spring Boot ledger service that transfers money between accounts and records each successful transfer for auditing.

The project focuses on atomic balance updates, idempotent requests, and concurrent access. It is intentionally scoped to a single service instance backed by one H2 database.

## Requirements and invariants

The service supports synchronous transfers between two existing accounts.

The following rules apply:

- source and destination accounts must exist and must be different;
- account IDs must be positive;
- the transfer amount must be positive, have at most two decimal places, and fit `NUMERIC(19,2)`;
- the source account must have sufficient funds;
- an account balance must never become negative;
- a successful request updates both balances and creates exactly one immutable `COMPLETED` transfer record;
- a rejected or rolled-back request does not create a ledger entry;
- an idempotency key must contain 1 to 100 characters and is globally unique.

Critical invariants are enforced both in the service and, where applicable, by database constraints.

| Constraint | Enforcement |
| --- | --- |
| Non-negative account balance | Service check and database check constraint |
| Different source and destination | Service check and database check constraint |
| Positive amount | Service check and database check constraint |
| Amount fits `NUMERIC(19,2)` | Service validation and database column definition |
| Unique idempotency key | Service behavior and database unique constraint |
| Atomic balance and audit updates | One database transaction |
| Concurrent account updates | JPA optimistic locking and retries |

## Assumptions and out of scope

The exercise makes the following assumptions:

- all accounts use the same currency;
- customers call this service directly;
- only one application instance and one SQL database are used;
- only completed transfers are part of the ledger;
- the client generates and retains the idempotency key;
- timestamps are generated by the application and represented as UTC instants.

The following concerns are out of scope:

- authentication and authorization;
- account creation and account-management APIs;
- currency conversion and multi-currency balances;
- cross-bank transfers;
- transfer cancellation or reversal;
- distributed deployment and external messaging.

## Running the application

### Prerequisites

- JDK 26
- no local Gradle installation is required; the Gradle wrapper is included

Run the tests:

```shell
./gradlew test
```

Start the service:

```shell
./gradlew bootRun
```

The API is then available at `http://localhost:8080`. Application data is stored in the file-backed H2 database at `./data/ledger`; Flyway applies the schema migration and Hibernate validates the resulting schema on startup.

### Creating development accounts

There is deliberately no account-management endpoint or seed data in this exercise. To try a transfer, open the H2 console at `http://localhost:8080/h2-console` and use:

```text
JDBC URL: jdbc:h2:file:./data/ledger;AUTO_SERVER=TRUE
User Name: sa
Password: <empty>
```

Create two accounts:

```sql
INSERT INTO account (balance, version, modified_at)
VALUES (100.00, 0, CURRENT_TIMESTAMP);

INSERT INTO account (balance, version, modified_at)
VALUES (25.00, 0, CURRENT_TIMESTAMP);
```

The generated IDs can be found with:

```sql
SELECT id, balance, version, modified_at FROM account ORDER BY id;
```

## API contract

### Transfer money

```http
POST /api/v1/transfer
```

The endpoint accepts query parameters:

| Parameter | Type | Rules |
| --- | --- | --- |
| `fromAccount` | integer | Positive ID of an existing source account |
| `toAccount` | integer | Positive ID of a different existing destination account |
| `amount` | decimal | Positive, at most two decimal places, fits `NUMERIC(19,2)` |
| `idempotencyKey` | string | Non-blank, 1-100 characters, globally unique per logical request |

Example request:

```shell
curl -X POST 'http://localhost:8080/api/v1/transfer?fromAccount=1&toAccount=2&amount=10.00&idempotencyKey=transfer-001'
```

A successful request returns `200 OK` and the persisted audit record:

```json
{
  "transferId": 1,
  "fromAccount": 1,
  "toAccount": 2,
  "amount": 10.00,
  "idempotencyKey": "transfer-001",
  "status": "COMPLETED",
  "createdAt": "2026-07-17T10:15:30.123456Z"
}
```

Errors use this shape:

```json
{
  "code": "INSUFFICIENT_FUNDS",
  "message": "Source account has insufficient funds"
}
```

| HTTP status | Error code | Meaning |
| --- | --- | --- |
| `400 Bad Request` | `INVALID_REQUEST` | A parameter is missing, malformed, or violates request constraints |
| `404 Not Found` | `ACCOUNT_NOT_FOUND` | One or both accounts do not exist |
| `409 Conflict` | `INSUFFICIENT_FUNDS` | The source balance is lower than the amount |
| `409 Conflict` | `IDEMPOTENCY_CONFLICT` | The key was already used with different transfer parameters |
| `409 Conflict` | `CONCURRENT_UPDATE` | Optimistic-lock retries were exhausted; the client may retry |

## Design and transaction semantics

### Domain model

`Account` contains:

- generated ID;
- `NUMERIC(19,2)` balance;
- optimistic-lock version;
- last-modified timestamp.

`MoneyTransfer` contains:

- generated transfer ID;
- source and destination account references;
- `NUMERIC(19,2)` amount;
- unique idempotency key;
- immutable `COMPLETED` status;
- creation timestamp.

### Transfer workflow

Each call runs in one database transaction:

1. Validate request parameters.
2. Look up an existing transfer by idempotency key.
3. Return that transfer if its source, destination, and amount match; otherwise reject key reuse.
4. Load both accounts in ascending account-ID order and verify they exist.
5. Verify that the source has sufficient funds.
6. Update the lower-ID account and flush it before updating the higher-ID account.
7. Persist the completed transfer audit record.
8. Commit both balance changes and the transfer atomically.

If any step fails, the transaction rolls back and no partial balance update or transfer record remains.

## Concurrency and idempotency

Accounts use JPA optimistic locking through a version column. The consistent account-ID update order also makes competing two-account operations follow the same write order.

Optimistic-lock and database-integrity conflicts are retried up to three total attempts with exponential backoff. Each attempt runs in a new transaction and reads the latest account and transfer state. If optimistic locking still fails, the API returns `CONCURRENT_UPDATE`.

Idempotency behavior is:

- the first successful request stores its key with the transfer;
- repeating the same key with the same source, destination, and numerically equal amount returns the original transfer without moving money again;
- repeating the key with different transfer parameters returns `IDEMPOTENCY_CONFLICT`;
- concurrent identical requests converge on the same committed transfer through the unique constraint and retry mechanism.

## Testing

The test suite includes:

- unit tests for transfer validation, insufficient funds, ordered updates, and idempotency;
- persistence integration tests for entity mappings and database constraints;
- controller tests for request validation and HTTP error mapping;
- Spring Boot integration tests for successful and rejected transfers;
- concurrent idempotent-request coverage that verifies money is moved once and only one audit record is created.

Tests use isolated in-memory H2 databases and do not modify the development database.

## Production considerations

This implementation is suitable for the stated coding-challenge scope, but a production service would normally require:

- a production-grade database such as PostgreSQL and a deployment-safe migration process;
- authentication, authorization, encrypted transport, and secret management;
- account and currency ownership rules;
- request-body DTOs with formal API documentation;
- metrics, tracing, structured audit logging, and operational health checks;
- a retention and reconciliation strategy for ledger data;
- load testing and a concurrency strategy validated against the production database;
- a defined approach to horizontal scaling and distributed coordination;
- explicit retry budgets, timeouts, and rate limiting.
